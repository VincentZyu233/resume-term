# Resume-Term 开发计划

## 目标
做一个跨平台桌面端工作区管理器，首版只支持 `Windows x64 / Linux x64`，用于保存并恢复类似 `tmux` 的窗口布局、终端命令和 SSH 会话。

## 已确定约束
- 技术栈：`Flutter + Rust`
- 配置格式：`YAML`
- 默认配置目录：`~/resume-term/config`
- 配置目录可在 GUI 中修改
- 首版不做 `tmux` 协议兼容，只做工作区编排与恢复
- 首版只验证 `Windows x64` 和 `Linux x64`

## 首版范围
- 保存 session / window / pane / split 结构
- 每个 pane 支持本地 shell 或 SSH 命令启动
- 每个 pane 支持默认启动命令、启动参数和可执行文件配置
- 支持恢复上次工作区布局
- 支持导入 / 导出 YAML 配置
- 支持配置目录切换
- 支持基础主题与常用快捷操作
- 顶栏提供 `Split` 按钮，支持 `上下 / 左右 / 自动`
- `自动` 分屏按当前窗格宽高比决定方向，默认规则为“宽大于高则左右拆，否则上下拆”
- `Split` 同时支持下拉菜单、快捷键和右键菜单入口
- 分屏行为只需要先保证 `Windows x64` 和 `Linux x64` 一致可用

## 架构方案
- `Flutter` 负责桌面 UI、布局编辑、配置编辑、状态展示
- `Rust` 负责进程管理、PTY / shell 启动、SSH 连接、底层会话恢复
- YAML 配置作为唯一的可编辑持久化格式
- 终端渲染层优先选成熟方案，首版目标是“能稳定显示和交互”，不是自研终端

## shell 策略
- Windows：优先 `pwsh`，其次 `powershell`，再其次 `cmd`
- Linux：优先 `$SHELL`，其次 `bash`、`zsh`、`sh`、`ash`
- 支持用户自定义 shell 优先级覆盖默认值
- 支持每个 profile 覆盖默认 shell / executable / command

## 默认启动命令
- 参考 Konsole 的思路，每个 pane 打开后可以直接执行一个默认命令，而不是只启动空 shell
- 参考 Windows Terminal 的 profile 设计，配置项需要支持直接指定可执行文件
- 默认命令优先级建议为：
  1. pane 显式配置的 `command`
  2. pane 对应 profile 配置的 `command`
  3. pane 对应 profile 配置的 `executable` + 参数
  4. 当前平台的默认 shell
- 默认命令用于 `ssh user@ip`、`htop`、`tail -f`、`top` 这类常见场景
- 若命令需要登录 shell 行为，则由 profile 显式声明，不靠 UI 猜测

## 配置模型
- `workspace`
  - `name`
  - `layout`
  - `panes`
  - `profiles`
  - `ssh_hosts`
  - `env`
- `pane`
  - `title`
  - `shell`
  - `command`
  - `working_dir`
  - `startup`
  - `split`
  - `executable`
  - `args`

- `profile`
  - `name`
  - `shell`
  - `executable`
  - `command`
  - `args`
  - `env`
  - `working_dir`

## UI 交互草案
- 顶栏保留一个主操作按钮，主入口为 `Split`
- 点击按钮直接执行上次使用的分屏方式
- 按钮下拉菜单提供：
  - `拆分视图 - 上下`
  - `拆分视图 - 左右`
  - `拆分视图 - 自动`
- `自动` 仅负责选择拆分方向，不改变当前 pane 内容
- 分屏后自动聚焦新 pane，并沿用当前 session 的配置上下文
- 支持在 pane 右键菜单中重复同样的分屏操作
- 支持在 pane 菜单里直接选择“启动命令 / 重启命令 / 切换 profile”
- 支持为常用命令提供快速入口，例如 SSH、监控、日志查看

## 开发里程碑
1. 先完成 Rust 核心：配置读写、会话模型、shell 启动、SSH 启动
2. 再完成 Flutter UI：布局树、session 编辑器、配置编辑器
3. 然后接通恢复流程：启动 app 后可重建上次 workspace
4. 补齐交互：Split 按钮、快捷键、右键菜单、自动分屏规则
5. 最后补齐体验：配置目录切换、导入导出、主题、启动偏好记忆

## GitHub Actions
- 使用 GitHub Actions 做编译和基础检查
- 使用 commit 信息控制 CI 触发，只有包含 `build action` 或 `build release` 的提交才进入指定构建 / 发布流程
- 至少包含：
  - `flutter pub get`
  - `flutter analyze`
  - Rust 相关检查与构建
  - Windows x64 / Linux x64 的桌面构建
- 产物命名统一带版本号，便于发布和回滚
- CI 以“可重复构建”为目标，后续再加 release 发布流程
- workflow 先以桌面端为主，优先保证 `Windows x64 / Linux x64` 的可编译性
- 后续如需发布，再增加打包和 release job，不影响当前开发计划

## 风险
- 终端与 PTY 在不同平台上的行为不一致
- SSH 会话恢复需要处理重连和断线场景
- 配置布局一旦复杂，编辑器需要避免过度抽象
- 分屏交互如果过早做得太像 tmux，可能会牺牲 GUI 的直观性
- 默认命令和默认 shell 的优先级如果不明确，后续会导致恢复行为混乱
- 暂不为 macOS / ARM64 设计首版分支逻辑，避免范围扩散
